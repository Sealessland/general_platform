# AI Native 开发工作流

## 基本原则

- AI 输出是草案，不是最终决策。
- 所有 AI 生成的代码、文档和测试都必须经过人工审查。
- 订单状态、库存、金额、权限、迁移等高风险改动必须人工复核。
- 只要 AI 的输出影响了设计、实现或验收，就要留下记录。

## 使用矩阵

| 场景 | AI 负责什么 | 人工负责什么 |
|---|---|---|
| 需求拆解 | 起草用户故事、边界和验收点 | 删除非 MVP 范围，补齐业务约束 |
| 技术设计 | 生成模块划分、状态流转、异常分支草案 | 明确事务边界、幂等、库存与退款规则 |
| API 设计 | 起草接口路径、请求体、响应体 | 统一错误语义、权限、分页和幂等约束 |
| 测试生成 | 枚举候选用例和边界输入 | 补齐并发、非法状态和金额边界 |
| Code Review | 指出潜在遗漏或不一致 | 决定是否修改，并补充验证 |
| 重构建议 | 给出拆分方向和依赖边界建议 | 保证外部行为不变，并同步文档 |

## 案例一：订单状态机

### 给 AI 的任务

为内容电商订单流设计状态机，覆盖 `CREATED`、`PAID`、`SHIPPED`、`FINISHED`、`CANCELLED`、`REFUNDING`、`REFUNDED`，说明合法流转、非法流转、库存释放节点、幂等要求与测试点。

### AI 输出的问题

- 漏掉了超时取消场景。
- 没有明确拒绝重复支付。
- 没把取消/退款和库存释放绑定起来。
- 没要求写订单事件日志。

### 人工修正

- 在 `backend/internal/order/domain` 中补了显式状态流转校验。
- 增加合法流转、非法流转、终态、重复支付相关测试。
- 补充订单状态机与库存锁相关 ADR。

## 案例二：MVP 可执行交付

### 给 AI 的任务

把 RedCart Copilot 的 MVP 拆成一个可运行的后端切面和一个边界清晰的前端交付简报，并显式保留幂等、状态流转和库存恢复规则。

### AI 输出的问题

- 倾向于默认 PostgreSQL 和 Redis 已经接入运行时。
- 前端容易滑向装饰性页面，而不是运营型工具界面。
- 默认的 Kimi worktree 流程依赖 Git 初始提交，这个仓库当时并不满足。

### 人工修正

- 运行时先使用内存适配层，把 PostgreSQL/Redis 保留为目标架构而不是硬前提。
- 后端补了服务层和 HTTP 层测试，覆盖下单、支付、发货、退款和库存恢复。
- 前端委托改为隔离目录执行，最后由主代理验收并回收结果。

## 案例三：风险审计修复

### 给 AI 的任务

根据测试审计中已经确认的风险点，按标准流程修复 HTTP 写操作 method gate、AI Copilot 权限隔离和 PostgreSQL 并发库存预锁问题，并补可执行回归测试。

### AI 输出的问题

- 初始修复容易只在 HTTP 层补判断，忽略服务层权限边界。
- 并发库存问题如果只在服务层读后写加判断，仍然无法阻止 PostgreSQL 并发超卖。
- 风险记录如果只保留在临时脚本或对话里，后续 agent 无法稳定追踪。

### 人工修正

- HTTP 状态变更路由显式要求 `POST`，并补 `GET` 不得触发写操作的回归测试。
- AI 生成入口要求商家角色，AI 任务读取按角色做所有权判断，避免 `merchant_id=0` 横向匹配。
- 新增 `SaveOrderWithInventoryLocks` 仓储契约，PostgreSQL 路径在事务内用条件更新原子预锁库存，内存路径在互斥锁内完成同样语义。
- 将已确认问题、修复状态、验证命令和剩余风险记录到 `docs/testing/2026-06-05-risk-audit.md`。
