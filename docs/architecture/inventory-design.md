# 库存设计

## 目标

- 在并发下单场景下避免超卖
- 保持 PostgreSQL 作为最终库存事实源
- 使用 Redis 处理高频预扣和幂等

## 当前 MVP 策略

1. 结算预览读取 SKU 与当前可售库存
2. 创建订单时必须提供 `Idempotency-Key`
3. 当前可执行 MVP 先使用内存适配层完成原子校验和预锁定
4. 下单时写入订单、订单明细和库存锁记录
5. 支付成功后确认扣减库存
6. 取消订单或退款完成后释放或恢复库存

## 下一阶段目标策略

1. Redis Lua 负责原子预扣库存
2. PostgreSQL 负责订单和库存锁的事务落盘
3. 超时取消、主动取消和退款统一走库存补偿路径

## 验证方式

- 单元测试覆盖库存不足与释放规则
- 服务层和 HTTP 层测试覆盖预览、下单、支付、取消、发货、退款与库存恢复
- 真实 Redis 适配层落地后，再补并发下单集成测试
- 迁移文件必须包含 `order_no` 和 `idempotency_key` 的唯一约束
