# 订单状态变更幂等

这份说明只覆盖当前高价值订单状态变更接口的幂等语义。

## 目标

- 让支付、发货、确认收货、退款申请、退款审批等高价值状态变更在重复请求下返回稳定结果
- 避免把“第一次已成功，第二次却报冲突”暴露给上层重试方
- 不改变订单、库存和事件写入的事实源，事实源仍然是 PostgreSQL

## 当前规则

### 1. 创建订单

- `POST /api/orders` 仍使用 `Idempotency-Key`
- 同一用户、同一幂等键重复创建订单时，返回同一笔订单

### 2. 已成功的状态变更重复调用

这些接口在目标状态已经达成时，直接返回当前订单视图：

- `POST /api/orders/{id}/pay`
- `POST /api/orders/{id}/cancel`
- `POST /api/orders/{id}/finish`
- `POST /api/orders/{id}/refund`
- `POST /api/merchant/orders/{id}/ship`
- `POST /api/merchant/orders/{id}/refund/approve`

返回口径保持不变：

- 消费者支付、取消、确认收货：`200`
- 发起退款：`202`
- 商家发货、退款审批：`200`

### 3. 仍然返回冲突的场景

如果目标状态尚未达成，但当前状态不允许这个动作，仍然返回业务冲突：

- 未支付订单发货
- 已支付订单取消
- 已完成订单退款
- 未进入 `REFUNDING` 的订单做退款审批

也就是说，只有“已经成功执行过同一动作”的重复请求才被视为幂等。

## 判定依据

幂等判断优先使用订单上的真实状态字段，而不是只看状态名：

- 支付：`paid_at` 或支付后状态
- 发货：`shipped_at` 或发货后状态
- 完成：`finished_at` 或完成状态
- 取消：`cancelled_at` 或取消状态
- 退款申请：`REFUNDING` / `REFUNDED`
- 退款审批：`REFUNDED`

这样可以避免把“已支付后直接退款，但从未发货”的订单误判成“已发货成功”。

## 非目标

- 不给这些接口补新的幂等键
- 不把商品创建、SKU 创建、登录、购物车加项这类写操作统一改成幂等
- 不改变订单状态机的合法迁移边界

## 完成证据

- 应用层测试覆盖重复成功请求返回当前订单视图
- HTTP 层测试覆盖重复成功请求返回稳定响应
- `bash scripts/validate-workspace.sh` 通过
