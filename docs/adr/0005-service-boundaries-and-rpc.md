# ADR 0005：服务边界与 RPC 使用场景

## 状态

已接受

## 背景

当前 RedCart Copilot 采用单体后端（Go + PostgreSQL/Redis）+ 独立 AI Service（Python）+ 静态前端的架构。随着订单量、商家数和 AI 功能增加，需要考虑什么时候把模块拆成独立服务、用什么协议通信。

## 决策

1. **优先保持模块化单体，直到出现明确的拆分驱动力**
   - 订单、库存、支付、行为事件目前共享强事务边界，拆分会引入分布式事务和最终一致性问题。
   - 当前性能瓶颈可通过 SQL 优化、批量写入、缓存解决，无需立刻服务化。

2. **以下场景优先使用 RPC 或独立服务**
   - **AI Copilot**：已经是独立进程/容器，适合用 HTTP/REST 或 gRPC 与后端通信；未来模型推理可能异地部署或扩缩容。
   - **支付/风控/清结算**：外部系统或专用团队，天然需要 RPC 接入，且通常有自己的 SLA 和可用性要求。
   - **通知/消息/行为分析**：可接受异步，优先用消息队列；如需同步确认再用 RPC。
   - **图片/视频/搜索等重算力服务**：与核心交易路径解耦，适合独立部署。

3. **暂不拆分为微服务的高内聚模块**
   - 订单、商品、库存、购物车、用户/商家认证：共享数据模型和事务，当前保留在同一进程内。
   - 这些模块通过 application/domain/infrastructure 分层解耦，未来拆分成本较低。

4. **协议选择**
   - 内部服务优先 **gRPC**：强 schema、二进制、双向流适合 AI/实时场景。
   - 对外和前端保持 **HTTP/REST/OpenAPI**：兼容性最好。
   - 异步事件优先 **消息队列**（如 NATS/Kafka/RabbitMQ），避免同步 RPC 级联阻塞。

## 影响

- AI Service 与后端之间的契约需要稳定化和版本化。
- 新增外部依赖服务时，必须先在 `docs/architecture.md` 中记录边界，再写适配器。
- 订单等核心模块继续通过数据库事务保证一致性，不盲目服务化。
- 监控和熔断应覆盖所有 RPC 出口，避免外部服务拖垮主交易路径。
