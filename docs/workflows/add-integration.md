# 新增集成或适配器

当你要接外部 API、数据库、缓存、模型 Provider、支付、消息队列或框架适配层时，走这个流程。

## 步骤

0. 先按 `git-worktree.md` 为当前集成任务创建独立 worktree，避免把中间件试验和主线开发混在一个工作区里。
1. 先识别集成需要满足的稳定契约。
2. 凭证、传输细节、重试、Provider 特有错误都留在集成层内部。
3. 暴露给上层的输入输出必须尽量中立，不把供应商类型直接泄漏出去。
4. 如果真实服务无法本地运行，至少补 mock、fixture 或契约测试。
5. 记录前置条件、限制和不支持的行为。
6. 如果 AI 参与了适配器方案、依赖选择、迁移脚本、测试或性能结论，按 `../../AI_WORKFLOW.md` 记录 AI 参与范围、人工修正和验证证据。
7. 中间件试验、性能对照和运行参数探索先隔离到独立分支；只有在基准、正确性和门禁结果稳定后，才整理成可合并提交。

## 不要默认假设

- 不要把供应商 SDK 类型直接变成领域类型，除非它本来就是公共契约。
- 不要让默认冒烟检查依赖外网。
- 不要把密钥、密码或凭证写进仓库文件。
- 不要在验证结论还不稳定时，把试验性中间件改动直接提交到当前主开发分支。
- 不要在同一个 worktree 里并行做多个中间件 spike。

## 完成证据

- `docs/architecture.md` 或更近的集成文档里，已经写清这个适配层边界。
- 本地验证覆盖了契约或 fixture 路径。
- AI 参与记录已经落到 `../../AI_WORKFLOW.md`、ADR、风险记录或 PR 描述中。
- `bash scripts/validate-workspace.sh` 通过。
